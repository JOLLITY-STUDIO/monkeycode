### 步骤 6：代码链路追踪 — 调色板/nametable 数据源定位

**输入**：
- `disasm/prg_annotated.txt`（44,323 行）
- `romdata/…_prg.bin`

**执行**：沿着 `$8297 加载调色板` → `$9085` → `$9B28` → `$9A71` 调用链，追踪数据如何流向 PPU

**输出** — 完整数据链路图：

```
标题画面:
  $00:8063  LDA #$0D         ; 调色板索引 #$0D
  $00:8063  JSR $8297        ; ─┐
  $00:8297  STA $E7          ;  │ 调色板索引存入 $E7
  $00:829D  LDA #$E5 / STA $4D  ; │ 数据指针低 = $E5
  $00:82A3  LDA #$00 / STA $4E  ; │ 数据指针高 = $00
  $00:82A5  JSR $9085        ; ─┤
  $00:9085  清空 $0467-$0566    ; │ 清空 VRAM 缓冲区
  $00:9095  LDA ($4D),Y      ;  │ 读取 data 块头部 (计数+类型)
  $00:90B4  LDA ($4D),Y      ;  │ 读取 bank 选择字节
  $00:90B7  CMP #$6D         ;  │ 如果 >= $6D →
  $00:90C2  JSR $C4B9        ;  │ 切换到 Bank $09 或 $0A
  $00:90DA  LDA ($92),Y      ;  │ 从 $A000+ 指针表取16位地址
  $00:90FB  LDA ($92),Y      ;  │ 读取数据块长度(N字节)
  ...                        ;  │ 循环读取 N 字节
  [data → $0568 OAM 缓冲区]  ;  │
  $00:9A71  (调色板写入PPU)  ; ←┘ $0568 → 查表 $9EA2 → PPU $3F00
```

**关键发现**：
1. **调色板 ROM 位置**：Bank $09/$0A 的 `$A000+` 区域，通过 `($4D),Y` 间接访问。索引 >= `$6D` 时自动切换到 Bank $09/$0A
2. **`$9EA2` 颜色映射表**：32 字节表，将索引色映射为 NES PPU 硬件颜色值
3. **Nametable 数据**：通过 `$8464` 场景初始化 → 脚本解释器 → `$992C`/`$98E8` → PPU `$2006/$2007`

**PPU 写入点汇总**：

| 地址 | 函数 | 功能 |
|------|------|------|
| `$00:98A0` | 清空 VRAM | `$2006=$2000`, 写 2048 字节 $00 |
| `$00:992C` | 直接 VRAM 写入 | `$E6/$E7` → `$2006`, data → `$2007` |
| `$00:9819` | 间接 VRAM 写入 | `($E6),Y` → `$2006`, `($E6),Y` → `$2007` |
| `$00:9A71` | 调色板写入 | `$0568-$0587` → `$9EA2` 查表 → `$3F00-$3F1F` |
| `$00:88CA` | 单 tile VRAM 写入 | `$8A14,Y` 查表 + `$05E8` 缓冲区 |
| `$00:8A91` | 批量 4 字节 VRAM 写入 | `($E8),Y` → ROM 数据块 |

#### 7.2 中断向量表

```
$0F:FFFA  .dw $C500  → NMI ($0F:C500)
$0F:FFFC  .dw $FFF0  → RESET ($0F:FFF0)
$0F:FFFE  .dw $C506  → IRQ ($0F:C506)
```

#### 7.3 场景/状态机 — `$1B` 游戏阶段 ID

RAM `$1B` 是核心状态变量（"游戏阶段/场景ID"）：

```
$00:804D  LDA $1B    → 读取当前场景
$00:804F  AND #$01   → 检查 bit 0
$00:8051  BNE $807A  → bit0=1 跳（已在场景中）
         → bit0=0 执行场景初始化

$01:851A  LDA $1B
$01:851C  ORA #$01   → 标记场景进入
$01:852A  AND #$FE   → 标记场景离开
```

分析得出的场景类型：
- `$1B` bit 0: 场景切换标志 (0=需要初始化, 1=已初始化)
- `$1B` bit 1-7: 场景编号 (0=启动, 1=标题, 2+=其他)
- `$1C`：场景子状态

#### 7.4 NMI/VBlank 流程（Bank $01 $8000-$8540）

```
NMI 触发 → $0F:C500
  ├→ 保存寄存器
  ├→ OAM DMA ($4014 = $02 → copy RAM $0200-$02FF)
  ├→ VRAM 缓冲区刷新
  │   ├→ 遍历 $05E8 队列（4 字节/条目：地址高+低+数据+标志）
  │   ├→ 将每个条目写入 PPU $2006/$2007
  │   └→ 队列结束标志 $00 → 退出
  ├→ Palette 更新（$2006=$3F00）
  ├→ 写入 PPU 控制寄存器 ($2000)
  ├→ 写入滚屏坐标 ($2005)
  ├→ MMC3 CHR bank 切换
  ├→ 读取手柄 ($4016/$4017)
  ├→ 比赛逻辑调用
  ├→ 音乐/音效更新 / MMC3 IRQ
  └→ 恢复寄存器 + RTI
```

#### 7.5 Boot/RESET 流程（Bank $0F + Bank $00）

```
RESET → $0F:FFF0
  ├→ SEI / CLD（禁用中断 / 清十进制）
  ├→ LDX #$FF / TXS（栈指针 = $FF）
  ├→ 设置 PPU: $2000 = $08, $2001 = $1E
  ├→ 清 RAM 页面 ($0000-$07FF)
  ├→ 等待 2 帧 VBlank
  ├→ MMC3 初始化
  └→ JMP $A200 → 进入 Bank $00 主循环

Bank $00 Boot ($8000-$8077):
  ├→ $8000-$800C: 函数跳转表 ($27 索引)
  ├→ $801F: 画面关闭 PPU OFF
  ├→ $8024: 场景初始化 ($8464)
  ├→ $8053: PPU 渲染关闭
  ├→ $805B: PPU 渲染开启
  ├→ $805E: 写入 VRAM 数据块
  ├→ $8063: 加载调色板 #$0D
  └→ $80A7-$80D3: 标题画面 — 等待按键
      ├→ $80E9: 场景=$01 → 比赛开始
      └→ $80C0: 光标闪烁逻辑
```

#### 7.9 数据加载体系（$9085 核心）

`$9085` 是 ROM 数据块加载器，被所有场景共用：

1. **参数**：`$4D/$4E` 指向数据块头部（2 字节：计数 + bank 选择）
2. **流程**：
   - 清空 $0467-$0566 缓冲区
   - 读取头部，$4D += 2
   - 循环：读取每个数据块描述符
     - 如果 bank 序号 >= `$6D` → 切换到 Bank $09/$0A
     - 否则在当前 bank 的 `$A000` 指针表查 16 位地址
     - `$92/$93` = 目标数据块地址
     - 拷贝 32 字节模板到 `$0568`（OAM 缓冲区）
     - 读取数据块长度 → 循环写入 `$0568` 缓冲区
3. **最终输出**：`$9A71` 将缓冲区数据通过 `$9EA2` 颜色映射表写入 PPU `$3F00`

### 步骤 8：动画/过场渲染管道 — 完整架构分析

---

#### 8.1 场景脚本解释器 `$84E7` — 操作码表

`$8464` 是场景入口（A=场景索引 → 查表 `$8AEE/$8AEC` → 定位 Bank + `$A000` 指针表 → 读取 2 字节脚本地址 → 设置 `$4D/$4E`），然后进入 `$84E7` 逐字节执行：

| 操作码 | 分派到 | 功能 |
|--------|--------|------|
| `$00–$D7` | `$84EF` | **写 tile 数据到 nametable**：从 `$52,$53` ROM 源逐 tile 通过 `$88CA` → VRAM 缓冲区队列 |
| `$D8–$DF` | `$8504` | **帧延时**：`(opcode - $D8)` → 查表 `$8AE6` 得延时帧数 → `JSR $9FA8` 等 VBlank |
| `$E0–$E7` | `$851C` | **设属性表位置**：`(opcode - $E1)` 取反后更新 `$53`（nametable 写指针），用于换行跳转 |
| **`$E8`** | `$8575` | **直接 VRAM 地址写入**：参数=PPU地址 → `$8920` 设 nametable 地址 |
| **`$E9`** | `$8580` | **等 2 帧 + 清输入状态** |
| **`$EA`** | `$858D` | **清屏 + 开 PPU + 填充双 nametable**：写 `$2000` 和 `$2400` 各 32×30=960 字节空白 tile → **场景切换黑屏过渡** |
| **`$EB`** | `$85C4` | **关 PPU + 动画资源回收** |
| **`$EC`** | `$85D2` | **条件判断**：读参数→如果不是 `$FF`，调用 `$89D2` 检查 `$1C` 场景子状态 |
| **`$ED`** | `$85EC` | **场景队列写**：找 `$0700-$0704` 空闲槽 → 写参数入队（多场景流水线调度） |
| **`$EE`** | `$8604` | **写 UI 面板**：PPU `$220B` 区域写 `$0B×$1E` 字节 → 状态栏 |
| **`$EF`** | `$8618` | **切换音效/音乐标记**：`$99` bit7 翻转 |
| **`$F0`** | `$862C` | **加载数据块**：读 2 字节参数 → 设为 `$50/$51` 新数据源 → 回 `$84E3` 继续批量写 tile |
| **`$F1`** | `$864A` | **写 UI 面板 2**：PPU `$220B` 区域另一段 |
| **`$F2`** | `$8659` | **播放音乐**：参数索引 → Bank $06 `$BB40` 音乐指针表 → `$97B6` 播放 |
| **`$F3`** | `$8678` | **设属性表地址高字节** |
| **`$F4`** | `$8682` | **等待输入**：参数=按键掩码(0=任意键, >0=指定键, <0=带等待, $FF=等待指定按键直到释放) |
| **`$F5`** | `$86B8` | **子操作码分发**：`$86C6` 表，含：画面闪烁/溶解效果、对话框渲染、文本滚动 |
| **`$F6`** | `$87B8` | **设精灵 Y 坐标**：`$4C` ← 参数 |
| **`$F7`** | `$87CB` | **帧延时（参数指定）** |
| **`$F8`** | `$87D9` | **滚屏开关切换**：`$7B` XOR $01 切换横屏滚屏 |
| **`$F9`** | `$87F8` | **进入比赛引擎**：切 Bank $02 → `$A212` 比赛逻辑（=场景→比赛的桥梁） |
| **`$FA/$FB`** | `$8814/$881B` | **播放音效 + 滚屏标记** |
| **`$FC`** | `$8831` | **加载调色板数据块**：通过 `$9085` 加载 → 回 `$84E7` 继续（=数据流式嵌入） |
| **`$FD`** | `$8837` | **滚屏推进**：`$51 += $40`（向下移动 64px 滚屏窗口）→ 实现平滑滚动 |
| **`$FE`** | `$8855` | **关 PPU + 延迟 + 场前进** |
| **`$FF`** | `$8862` | **场景结束/跳转**：读参数 → 可能是场景链式跳转 |

#### 8.2 渲染管道全景图

```
ROM 场景脚本 bytecode
  │
  ▼
$8464 场景入口 (A=场景索引)
  ├→ 查表 $8AEE/$8AEC → 定位脚本所在 Bank + $A000 偏移
  ├→ 读 2字节指针 → 得到 bytecode 真实 ROM 地址
  └→ 设 $4D/$4E 为脚本指针
  │
  ▼
$84E7 脚本解释器 (主循环)
  ├→ LDA ($4D),Y 读操作码
  ├→ 按操作码范围分发
  │
  ├── < $D8: 写 tile ──→ $88CA → VRAM缓冲区队列($05E8+)
  ├── $D8-$DF: 等帧 ──→ $9FA8 等 VBlank
  ├── $E8-$FF: 控制操作码
  │
  └→ $8879: 递增 $4D/$4E → 读下一条操作码
  │
  ▼
NMI 帧中断 (Bank $01:8000) 每帧触发
  ├→ OAM DMA ($4014)
  ├→ 遍历 $05E8 队列 (4字节/项)
  │   ├→ 控制字节 (bit7=是否设置地址, bit6=横向/纵向)
  │   ├→ PPU地址高 + 低 → $2006
  │   └→ 数据字节 → $2007
  ├→ 调色板 → $3F00
  ├→ PPU控制 ($2000) + 滚屏 ($2005) + MMC3 CHR切换
  └→ 手柄轮询 ($4016/$4017)
  │
  ▼
Bank $0F 动画效果引擎 (30+ 种，$C500-$D000 区域)
  ├→ $C50C: 精灵渲染/动画 (从 ROM 数据块绘制球员/物体)
  ├→ $C53C: 球员效果 (属性相关的动画)
  ├→ $C56C: 过场效果 (场景过渡动画播放)
  ├→ $C578: 过场效果 (另一种过渡)
  ├→ $C4B2/C4B9: MMC3 bank 切换封装
  └→ 其他 25+ 种效果函数
```

#### 8.3 过场动画的"丰富感"来自哪里

**1. 逐 tile 逐帧控制**
- 脚本可以逐帧往 nametable 写 tile，而不是一次性填充
- `$D8-$DF` 操作码精确等待 1–8 帧，实现逐帧动画
- 比如：写一排 tile → 等 2 帧 → 写下一排 → 效果就是"从左到右展开"

**2. 滚屏操控**
- `$F8` 开关滚屏、`$FD` 推进滚屏偏移
- `$01:8086` NMI 里每帧写 `$2005` 滚屏寄存器
- 实现场景中画面平移/震动/镜头跟随

**3. CHR bank 切换**
- `$01:80AF-$80D4` NMI 每帧切换 MMC3 CHR 寄存器（`$8000/$8001` bank 2-5）
- 不同 bank 里存不同动画帧的 tile → 实现精灵动画翻页

**4. 调色板动态加载**
- `$FC` 操作码通过 `$9085` 热加载新调色板
- 实现场景间的颜色过渡、闪白、淡入淡出

**5. Bank $0F 的 30+ 种"画面效果/动画"函数**
- 不是单一效果，而是一个函数表驱动的大系统
- 对应控球、射门、传球、碰撞等不同比赛事件的不同视觉反馈

**6. 场景队列（`$0700-$0704`）**
- `$ED` 操作码写场景队列
- 支持流水线式场景调度 → 当前场景播放时，下一个场景已预加载

#### 8.4 游戏实际启动流程

```
RESET ($0F:FFF0)
  ├→ PPU初始化、清RAM、MMC3初始化
  └→ JMP $A200 → Bank $00 主循环入口
  │
  ▼
$27 = 0: 初始场景
  ├→ $8464 场景 #xx (从 $83DC[$26] 查表)
  ├→ $82B5 初始化场景变量
  ├→ $C56C 播放过场动画效果
  └→ 检查 $26 vs $E4 是否要加载新场景
  │
  ▼
$27 = 1,2,3,4: 依次推进关卡/过场
  ├→ 每阶段查 $83DC/$83FE/$8420[$26] 表
  ├→ 决定是否需要 $C56C/$C578 过场效果
  └→ 最终到达标题画面 (Title)
  │
  ▼
标题画面循环 ($80A2-$80D3)
  ├→ 等按键
  ├→ Start → 场景切换 ($80E9: $8464 场景 #$01)
  └→ $C578 过渡效果 → 进入比赛引擎
```

**关键**：`$26` 是"游戏总进度"计数器，不是简单的难度值。`$26` 的每个值对应一个宏观阶段（开场动画1→开场动画2→标题→选队→比赛…），查 `$83DC`、`$83FE`、`$8420` 等 ROM 表获取对应的场景脚本索引。

### 10.1 球员属性指针表 — Bank $0C `$8DC2`

**ROM 绝对偏移**: `$30DC2`  
**索引方式**: `X = (球员_ID >> 2) * 2` → `LDA $8DC2,X`: `LDA $8DC3,X`  
**含义**: ~64 个 2 字节指针条目 (对应 256 ID / 4)

**`$8DC2` 原始 hex**:

| X | 字节对 | 指针 | X | 字节对 | 指针 |
|---|---|---|---|---|---|
| 8 | `E0 8D` | `$8DE0` | 19 | `40 91` | `$9140` |

| 10 | `94 8E` | `$8E94` | 21 | `40 91` | `$9140` |
| 12 | `6B 8F` | `$8F6B` | 22 | `B0 91` | `$91B0` |

### 10.2 属性加载代码 — Bank $0C `$8248`

**ROM 绝对偏移**: `$30248`  
**两级查表流程**:

```asm
;$8248: 第一级 — class/type 查表 ($86B8)
       LDA $xxxx    ; 球员 ID
       LSR A, LSR A, TAX          ; X = ID >> 2 (~64 种 class)
       LDA $86B8,X  → STA $05E6   ; class/type 标记

;$8252: 第二级 — 属性指针查表 ($8DC2)
       TXA                        ; X = ID >> 2
       ASL A, PHA, TAX            ; X = (ID >> 2) * 2 → 索引 $8DC2
       LDA $8DC2,X → STA $61     ; 属性块地址低字节
       LDA $8DC3,X → STA $62     ; 属性块地址高字节

;$8260: 读取属性块头部 (3 bytes)
       LDY #$00
       LDA ($61),Y → PHA          ; byte 0 → push (type/level?)
       INY
       LDA ($61),Y → PHA          ; byte 1 → push
       INY
       LDA ($61),Y → STA $05E7   ; byte 2 → 状态变量
       INY
       LDA #$06 → STA $05E8      ; 循环计数器 = 6
       ; ... VBlank 等待 + bytecode decode ...
```

**`$86B8` table** — class/type lookup (Bank $0C, ROM offset $30CB8):
```
$86B8: AF 2E  34 33  AF 2E  34 34    ; 前 4 条含零页地址? ($2EAF, $3334)
$86C8: 0E 0E  0E 00  0E 0E  0E 0E    ; 
$86D0: 0E 12  12 12  12 12  12 00    ; tile ID 范围 0E~12
$86D8: 01 01  01 02  00 02  00 00    ; 
$86E0: 01 01  01 02  00 02  00 03    ; level 范围 01~03
$86E8: 01 02  05 05  05 06  04 06    ; 
$86F0: 05 05  05 06  04 06  00 00    ; 
```
→ `$86B8` 每 `(ID>>2)` 对应 1 字节 class/type 值  
→ `$86C8` 起可能是 tile/level 数据

### 10.4 对手阵容/Roster 数据表 — Bank $02 `$AA47`

**来源**: `_tmp_bzk_out/bank_02.asm`, `_tmp_search_player.mjs`  
**ROM 位置**: Bank $02, CPU `$AA47` = PRG offset `0x4A47`（`$8000-$9FFF` 映射到 PRG `$4000-$5FFF`）  
**代码入口**: `$8855`（主流程）→ 根据 `$26`（当前对手索引）选择 Group → `$88B7`（加载场上 11 人）→ 可选 `$8893`（加载 bench）

#### 分组逻辑（`$8855`）

```
$8855: LDA $E4         ; 当前游戏进度
       CMP $26         ; $26 = 当前对手索引
       BCS $88A8       ; 若 $E4 >= $26 → 跳过加载（阵容已存在）
       
       LDA $26
       BEQ $887C       ; S0      → X=$00, Group 0
       CMP #$06
       BEQ $8884       ; S6      → X=$0C, Group 1
       CMP #$0C
       BEQ $887C       ; S12     → X=$00, Group 0（复用）
       CMP #$10
       BEQ $888C       ; S16     → X=$18, Group 2 ★
       JMP $A8A8       ; 其他   → 只设 ACE 参数
```

**三层分组**：

| 分组 | X offset | 覆盖对手 | 特点 |
|------|---------|---------|------|
| **Group 0** | `$00` | S0-S5, S12-S15 | 早期 + 中期复用 |
| **Group 1** | `$0C` | S6-S11 | 中期 |
| **Group 2** | `$18` | **S16+** | 后期（全国大赛起），**额外加载 bench** |

#### `$AA47` 表 — 完整 Overlapping 布局

ROM 数据（小端序, Bank $02, PRG `0x4A47`）：

```
+$00-$0A:  02 03 04 05 06 07 08 09 0A 01 0B    ← G0 场上 11 人
+$0B:      00                                     ← G0 → $2C (ctrl)
+$0C-$16:  0F 0D 0E 14 10 0C 13 12 15 11 16    ← G1 场上 11 人
+$17:      00                                     ← G1 → $2C (ctrl)
+$18-$22:  22 1B 1C 14 1D 17 18 11 1A 01 15    ← G2 场上 11 人
+$23-$2C:  19 1F 10 12 13 16 1E 20 21 0F       ← G2 bench 10 人 → $0408
+$2D:      01                                     ← G2 → $2C (ctrl)
+$2E+:     ACE 表（$AA75），每对手索引 1 byte
```

> **Overlapping 原理**：`+$0C` 既是 G0 的 ctrl byte 也是 G1 的球员[0]；`+$18` 既是 G1 的 ctrl byte 也是 G2 的球员[0]。通过 X offset 区分读取语义，节省 ROM。

#### `$88B7` 子程序 — 加载场上 11 人

```asm
$88B7: LDA #$0B       ; 循环 11 次（0-based counter）
       STA $ED
       LDY #$00
$88BD: LDA $AA47,X     ; 从 ROM 读球员 ID
       STA $0300,Y     ; 存到对手 RAM（间隔 $0C 字节）
       INX
       TYA: ADC #$0C: TAY
       CMP #$84        ; Y=120 后退出 → 正好 11 次
       BCC $88BD
       RTS              ; X 已前进 11
```

- 加载 **11 个**对手球员 ID（非 12 个）
- 存入 `$0300`, `$030C`, `$0318`, ... `$0378`
- **第 12 个 byte**（`$AA47+X_offset+11`）存到 `$2C`，是**阵型/战术控制字节**，不是球员 ID

#### `$8893` Bench 加载 — 仅 Group 2 (S16+)

```asm
$8891: LDY #$00
$8893: LDA $AA47,X     ; X 起始 = $23
       STA $0408,Y     ; 存到 $0408 区域，间隔 4 bytes
       INX
       TYA: ADC #$04: TAY
       CMP #$28        ; 10 次后退出
       BCC $8893
```

- Group 0/1 **跳过此段代码**（走 `JMP $A8A3`）
- Group 2 **额外加载 10 个 bench 球员 ID** 到 `$0408` 区域
- RAM 分布：`$0408`, `$040C`, `$0410`, ..., `$042C`（10 slots）
- 这就是 S16+ **替补席系统的数据源**

#### ACE 球员表 — `$AA75` (= `$AA47 + $2E`)

```
S0-S5, S12-S15:  ACE=$00
S6-S11:          ACE=$01
S16-S22:         ACE=$02
```

每个对手有一个 ACE/王牌球员标记，不同阶段不同值。

#### 关键 RAM 映射

| RAM 地址 | 大小 | 内容 |
|----------|------|------|
| `$0300` | 11 slots × $0C | **对手场上 11 人**（加载自 `$88B7`） |
| `$0408` | 10 slots × $04 | **对手 bench 10 人**（仅 S16+ 加载自 `$8893`） |
| `$0468` | 11 slots × $04 | **玩家自己队伍 11 人**（游戏初始化时加载） |
| `$2A` | 1 byte | ACE 球员标记（来自 `$AA75`） |
| `$2C` | 1 byte | 阵型/战术控制字节 |
| `$26` | 1 byte | **当前对手索引**（关卡进度标识） |
| `$E4` | 1 byte | **已解锁进度**（存档持久化） |

#### 替补系统激活条件

- **S0-S15**（全国大赛前）：无 bench 数据，对手固定 11 人，只可调位置。半场换人（如有）是**事件脚本硬编码**（查 `$83DC` 事件表触发），非用户可控
- **S16+**（全国大赛起）：`$8893` 加载 10 个 bench 球员 → 替补席 UI 可用 → 玩家和对手都可换人

### 代码逻辑分析

场景系统位于 bank_00 `$8464`+：
1. `sceneInit($8464)` — 查 `$8AEC` bank 表获取场景脚本 bank，switch reg 6

6. 操作码 `$00-$D7`（`$84EF`）— tile 写入，从 `($53,$52)` 读取→写 VRAM 缓冲区 $05E8，INC $53

**$F0 处理机详细（$862C）：**

```asm
862F: STA $4F          ; scrollBase = paramLo
8631: STA $51          ; dataSrcHi = paramLo
8633: INY

863A: LDA $4D          ; 脚本指针 += 3（跳过 F0 + 2 参数）
8647: JMP $84E3        ; → $84E3: LDA $51; STA $53; 继续解释
```

**$84E3 重入点 / $84EF tile 写入：**

### 直接引用路径：`$8EF0-$8FCB` tile 上传函数

在 `bank_00.asm` 中找到**直接引用 bank_08 的代码路径**。

**关键常量确认**（`_chr_bank_map.md` line 187）：
- `$C4B9`（MMC3 切换例程）默认切换 **reg 7**（`$A000-$BFFF`），非 reg 6
- 调用约定：`LDX #bankNum; JSR $C4B9`

**`$8EF0-$8FCB` 完整流程：**

```
入口: A = tile 索引参数
      $5B bit0 → $EB (高字节初值)
      $5C/$5D → $67/$68 (PPU nametable 坐标)

$8EF0: TAX              ; 保存 A→X
$8EF1-$8F1E: ×17 乘法运算
      $EA = index * 17 (低字节)
      $EB = bit0($5B) + carry (高字节初值)
$8F20: LDA $EB
$8F22: ADC #$A0          ; +$A0 = 基址 $A000
$8F24: STA $EB           ; 指针 = $A000 + index*17 + bit0*256

$8F26: LDX #$08          ; bank 08
$8F28: JSR $C4B9          ; 切换 reg 7 → bank_08

$8F2D: LDA ($EA),Y       ; 从 bank_08 (reg 7) 读取第1字节
$8F2F: STA $E7           ; → PPU 地址高字节
$8F31: JSR $8FD1          ; PPU 地址处理

; 4 轮循环 (每轮 4 字节 tile 数据)
$8F3A: LDA #4; STA $E8   ; 4 轮
$8F3E: LDY $67; LDX $68  ; PPU nametable 坐标
$8F44: JSR $9B28          ; 初始化 VRAM 队列条目
$8F49: LDA ($EA),Y        ; 从 bank_08 读 4 字节
$8F4B: STA $05E8,X        ; → VRAM 队列
$8F4E-$8F52: loop Y=0..3
$8F54: JSR $9B5E          ; 提交队列条目
$8F57-$8F6F: 循环4轮, 每轮后 $67+=0x20

$8FCB: LDX #$07           ; 恢复
$8FCD: JSR $C4B9          ; reg 7 → bank_07
$8FD0: RTS
```

**数据访问方式**：
- bank_08 映射到 reg 7（$A000-$BFFF），通过 `($EA),Y` ZP 间接寻址
- meeting tile 数据在 bank_08 offset $4A0 → reg 7 地址 = $A4A0
- 指针计算：`$A000 + index*17 + bit0($5B)*256`

**PPU 上传链**：
```
$8F26 LDX #08; JSR $C4B9  → reg 7 = bank_08
$8F2D LDA ($EA),Y          → 从 $A4A0+ 读 tile 数据
$8F4B STA $05E8,X          → 写入 VRAM 队列
$866B JSR $97B6            → NMI 时 flush 队列到 PPU VRAM
```

**与 TS 代码的对应**：
- `SceneInterpreter.ts` 的 `writeTileToVram()`（line 576）目前是空实现/stub——**已完成** ✅
- `CoreUtils.ts` 的 `switchMmc3Bank()`（line 288）翻译自 `$C4B9`——但 TS 版增加了 reg 参数
- `NMIHandler.ts` 的 VRAM queue flush（line 42-83）翻译自 `$97B6/$982F`——已实现

### 一、Nametable 数据位置

#### 1. 主要 Nametable 加载器 — Bank 11

- **romBase**: `0x16010`
- **CPU 入口**: `$800C` (`loadNametable`)、`$8083` (`loadAttributes`)

**Nametable tilemap 数据：**

| ntBank | CPU 地址 | ROM 偏移范围 | 大小 |
|--------|----------|-------------|------|
| 0 | `$800C` | `0x1601C` - `0x163DB` | 960 B |
| 1 | `$810C` | `0x1611C` - `0x164DB` | 960 B |
| 2 | `$820C` | `0x1621C` - `0x165DB` | 960 B |
| ... | ... | ... | ... |
| *N* | `$800C + N×0x100` | `0x1601C + N×0x100` | 960 B |

> 加载逻辑：`ntBank = RAM[$5A]`, 源起始 = `0x800C - 0x8000 + ntBank * 0x100`。每个 nametable 为 30×32 = 960 字节的 tile 索引。

**Nametable attribute 数据：**

| CPU 地址 | ROM 偏移范围 | 大小 |
|----------|-------------|------|
| `$8083` | `0x16093` - `0x160D2` | 64 B |

> 64 字节属性表，写入 PPU `$2007` 属性表区域。

#### 2. Meeting 场景固定 Nametable — Bank 08

| CPU 地址 | ROM 偏移范围 | 大小 | 用途 |
|----------|-------------|------|------|
| `$84A0-$854F` | `0x104B0` - `0x1055F` | 176 B (11×16 tile) | Meeting 屏 tilemap |

> 在 `meeting_ppu_data.py` 中被硬编码提取。通过 `$8EF0-$8FCB` 函数以 bank_08 reg7 映射访问（地址 = `$A000 + index*17 + bit0*256`）。

#### 3. 场景脚本驱动的 Nametable — Bank $09 / $10

- **Bank 09**: romBase `0x12010` - `0x1400F` (8KB, 全是 data)
- **Bank 10**: romBase `0x14010` - `0x1600F` (8KB, 全是 data)

**`$A000+` 指针表**: 场景脚本 bytecode 地址表（ROM `0x12000+` 即 Bank 09 的 offset `0x0000`+ → 实际 ROM `0x12010+`）。
- 场景索引 ≥ `$6D` 时，`$9085` 数据加载器自动切换到 Bank $09/$0A
- `$862C` 操作码 `$F0` 加载 nametable 数据块指针 → 指向 Bank $09/$10 内嵌套的 tile 数据
- 每个场景脚本包含多段 nametable 写入序列（逐 tile 通过 VRAM 队列写 PPU `$2007`）

**结构**：Bank $09 offset [0x0000-0x1FFF] = ROM [0x12010-0x1400F]，含：
- `$0000+` (ROM `0x12010+`): 场景脚本指针表
- `$0XXX+` (ROM `0x12XXX+`): 各场景的 tile 数据块 (960B nametable + 64B attribute)

### 二、Palette 数据位置

#### 1. NES 颜色映射表 — Bank 00

| CPU 地址 | ROM 偏移 | 大小 | 用途 |
|----------|----------|------|------|
| `$9EA2` | `0x1EA2` - `0x1EC1` | 32 B | 索引色 → NES PPU 硬件色值映射 |

> 位于 `$00:9A71` 调色板写入函数中，将 32 字节缓冲区 `$0568-$0587` 通过查表映射为 NES 64 色调色板编号。

#### 2. 运行时调色板数据 — Bank $09 / $10

- **入口**: Bank 00 `$9085` 数据块加载器
- **参数**: 调色板索引存入 `$E7`（例如标题画面 `LDA #$0D`）
- **加载逻辑**: 索引 >= `$6D` 时切换 reg 7 → Bank $09 或 $0A，从 `$A000+` 区域读取

| 调色板索引示例 | 对应 ROM | 用途 |
|--------------|----------|------|
| `#$0D` | Bank 09/0A `$A000+` 指针→数据块 | 标题画面主调色板 |
| 其他 (0x6D+) | Bank 09/0A 对应索引 | 各场景/sub-palette |

> 最终写入目标：PPU `$3F00-$3F1F`（32 字节，含背景+精灵调色板）。

#### 3. 场景脚本内嵌调色板（`$FC` 操作码）

- 场景脚本通过操作码 `$FC` (`loadPaletteBlock`, `$8831`) 热加载新调色板
- 数据同样从 Bank $09/$10 的 `$A000+` 指针表定位

## 2026-07-19 — 48 Bank 完整细分分类（按子系统/游戏功能）

### 目的

不再用 "bank_01" 这种粗糙标签，而是按 ROM 内实际数据用途，将每个 bank 内的 data 段标注为具体的游戏功能区域（标题画面 NT、比赛精灵元数据、球员属性表、场景脚本 bytecode 等）。

### 分类标准

- **Code bank**: 含可执行代码 + 内嵌数据表（如跳转表、场景映射表、trig 表等）
- **Data bank**: codeCount=0 的纯数据 bank，按用途标注
- **Code+Data bank**: codeCount>0 但 data 占多数的 bank
- **CHR bank**: PPU pattern table 数据（tile 图形）

```
  ▼

  ▼
```

- Display list is the ONLY path for PPU data: nametable, palette, CHR pattern all go through it
- No display list = zero visual data enters PPU pipeline

```

            UNLOADED         UNLOADED    dl empty
            $C503 stub       scene scr.  scenePreProcess
```
