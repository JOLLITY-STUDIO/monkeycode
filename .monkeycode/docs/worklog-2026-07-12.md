### 关键发现: 角色初始值数据流 (07-13)

**之前的误判**: "角色初始值不存为独立表, 由场景字节码通过脚本解释器动态写入"
**纠正**: 角色初始CLASS_ID存储在 ROM **0x05E64A** (10角色×0x18字节), 加载流程如下:

```
ROM 0x05E64A (角色能力初始表, 0x18B×10角色)
      │
      │ FUN_00010a94: 每角色复制 0x0E 字节 + 补 0x0A 零 → 共 0x18 字节
      ▼
RAM 0xFFFFA4CC (DAT_ffffa4cc, 10角色能力运行时表)
  offset +0x10 = DAT_ffffa4dc = 初始 CLASS_ID
      │
      │ FUN_00010bbe: 读取 CLASS_ID → 写 0xFF603C 槽 offset 0x0B
      ├── 同时从 ROM 0x05EDE9 读职业基础 AT/DF (class_id×0x1C 索引)
      └── 同时从 ROM 0x082A59 读角色 AT/DF 修正 (character_id×6 索引)
      ▼
RAM 0xFF603C (战场角色槽, 0x60B×20槽)
      │
      │ FUN_0000e2da / FUN_00011c78: 角色升级/转职后回写
      ▼
RAM 0xFFFFA4CC ← 更新后的能力值(同步)
```

**新增 ROM 地址**:
| 地址 | 大小 | 内容 |
|------|------|------|
| `0x05E64A` | 0xF0 | 角色能力初始表 (10×0x18字节, 含CLASS_ID于+0x10) |
| `0x05EDE9` | ~0x1B0 | 职业基础AT/DF属性表 (0x1C字节/职业, class_id×0x1C索引) |
| `0x082A59` | 0x3C | 角色AT/DF修正表 (6字节/角色) |

**关键 Ghidra 函数**:
| 函数 | ROM地址 | 功能 |
|------|--------|------|
| `FUN_0000c80c` | 0x00C80C | 游戏主初始化, 调用 FUN_00010a94 |
| `FUN_00010a94` | 0x010A94 | 从 ROM 0x05E64A 加载10角色能力表到 RAM |
| `FUN_00010bbe` | 0x010BBE | 玩家角色初始化 (读DAT_ffffa4cc+ROM表→写0xFF603C槽) |
| `FUN_0000e2da` | 0x00E2DA | 角色RAM槽→能力表回写(同步) |
| `FUN_00011c78` | 0x011C78 | 同上, 第二处回写点 |

### 场景加载调用链 (07-12)

```
VBLANK (0x08294, 主循环)
  ├── FUN_0000870a
  ├── FUN_00009074
  ├── FUN_00009400 → FUN_00000270
  └── FUN_00008a6c / FUN_00008a20

FUN_00009ec4 (地图加载)
  ├── 从 0x061CB0 读地图指针 (DAT_ffffa49c=场景索引-1)
  ├── 从 0x061E24/0x061E28 读 Tile 重映射表
  ├── 从 0x061D2C/0x061D30 读 Tile 属性表
  └── FUN_0000c7ae (后处理)

FUN_00009fc2 (场景初始化)
  ├── FUN_000099b2 (×2)
  ├── FUN_00009ffe (单位脚本加载)
  │     - 场景11: 0x080674 (DAT_ffffa000=1)
  │     - 场景12/23: 0x080328 (DAT_ffffa000=0)
  │     - 其他: DAT_ffffa000=0xFF (无特殊脚本)
  │     - FUN_0000a052 (DMA精灵数据, 0x11字节/条目)
  └── FUN_0000a0ac (设置渲染参数)

FUN_0000a122 (场景单位配置拷贝)
  └── 从 0x0821BE[scenario] 读 128字节(32dword) → 0xFFFF9F62

FUN_0000a16a (角色槽位初始化)
  ├── 从 0x05E5D8 表读角色RAM地址
  └── 检查 byte[0x20] 决定渲染/逻辑分支

FUN_00009498 (字节码解释器)
  └── 0x94C2=循环出口, 按 DAT_ffff95b8 跳转表调度脚本指令
```

### 角色 RAM 完整结构 (0x60=96字节/槽, 0xFF603C基址)

```
byte[0x00]: 角色基础ID        byte[0x01]: 战场编号
byte[0x08]: 状态标志          byte[0x09]: AT成长率
byte[0x0A]: DF成长率          byte[0x0B]: CLASS_ID ★
byte[0x0C]: 道具1 ID         byte[0x0D]: 道具1 等级
byte[0x14]: 等级              byte[0x16]: X坐标
byte[0x17]: Y坐标              byte[0x20]: unit type/阵营
byte[0x21]: 移动类型           byte[0x2C-2D]: 佣兵数
byte[0x2E-2F]: AT修正          byte[0x38]: 当前AT
byte[0x39]: 基础AT            byte[0x3A-3B]: DF修正
byte[0x50-53]: 魔法槽          byte[0x5E]: 经验
byte[0x5F]: 命令列表
每角色含8个佣兵子槽(各12B) + 5个道具槽(各0xC偏移)
```

### FFFF 分隔组格式 (单位数据)

```
[word, word, ..., word, TERM]
TERM ∈ {0xFFFF=标准单位, 0xFFF9=职业序列0..11, 0xFFFB/C/A/E=特殊标记}
0x097380 → 指针表(31条目) → 0x0975F4(基础组,滑动窗口) / 0x097400(场景组,49组)
```

### 关键发现

**地图指针修复**: `md-map.js` 的 `readU32` 原使用 24-bit 小端掩码, 修正为 32-bit 大端(68K原生格式)。31张地图尺寸已验证 (16×44 ~ 48×64)。

**角色名称定位**: ROM 中无独立角色名表。
- 名称表 `0x0618E8` (80条) 为职业名, 非角色名
- 角色名散落在场景脚本中, 使用 VRAM 字模 tile 索引编码 (0xB0-0xDF 范围)
- 需提取 VRAM 字体表才能解码为可读文字
- 英文脚本 (`767895e3-script-1.txt`, Hiryuu Honyaku 翻译) 含 35 位有名角色

**场景数据四层结构**:
1. 地图 tile (0x061F1C 起, w×h 字节)
2. Tile 属性 (0x061D2C/0x061D30 双指针, 各 0x280 字节)
3. Tile 重映射 (0x061E24/0x061E28 双指针, 各 0x100 字节)
4. 单位脚本 (尚未定位固定格式表)

**道具表验证**: 36 条目, 首个 0xFF 字节需跳过
